What it is

Tell it the goal. It runs the agent until the goal is really met.

You point goaly at a goal and a way to check it. It runs your coding agent over and over — feeding back what's still wrong each time — and only stops when the work passes a fixed bar. The trick is that bar can't move: the agent can't quietly weaken its own test to "finish".

1 Freeze the bar

The success check is written once, hashed, and frozen before the loop starts. No later step can rewrite it — so "until it's done" can't collapse into "until the agent lowers the bar".

2 Two keys for DONE

A run is done only when both turn: the frozen check passes and an independent reviewer doesn't veto. Gaming one is possible; getting both is much harder.

3 Fail closed

Anything that errors, crashes, or returns garbage becomes a fail — never a false green. A broken check can never accidentally mark the run complete.

That's the whole point: correctness you can trust when the agent runs unattended. The control loop itself makes zero LLM calls — it's a small, deterministic state machine — so every run is replayable and resumable from a log. See how the loop works ↓

Get started

From install to your first verified run in a few minutes.

Before you start: Node ≥ 20 and git, plus at least one agent CLI on your PATH — claude, codex, droid, or pi. (No CLI yet? Use --harness fake to try the loop with zero network.)

01
Install it

Builds dist/ and puts goaly on your PATH.

# clone, then from the repo root:
make install        # == npm install -g .
goaly help
02
Run against a check you already have

Point at an existing test command. goaly freezes the contract, asks you to approve it once at Seal, then loops until npm test passes and the reviewer agrees.

goaly run --goal "make the parser handle empty input" \
  --verify-cmd "npm test"
03
Or just give it a goal — the agent writes the check too

The goal is a positional and run is optional. With no --verify-cmd, the LLM authors the verification (--generate) and checks the work, using Claude — all defaults. Add -d (alias --defaults) to run hands-off: it auto-accepts the Seal pause (the contract is still frozen and logged loudly).

# approve the contract once at Seal
goaly "add a /health endpoint returning 200"

# …or fully hands-off
goaly -d "add a /health endpoint returning 200"

A few more things you'll reach for

# pick a harness, cap iterations, set a token budget
goaly run --goal "…" --harness codex \
  --max-iterations 8 --budget-tokens 500000

# a long goal from a file (or stdin)
goaly run --goal-file ./GOAL.md --generate
cat GOAL.md | goaly run --goal - --generate

# resume a crashed run from its log
goaly run --goal "…" --resume run-<id>
# watch the agent's turns live as they happen
goaly run --goal "…" --verify-cmd "npm test" --stream

# jail the agent + verifier in an OS sandbox (opt-in)
goaly run --goal "…" --verify-cmd "npm test" --sandbox

# inspect past runs (read-only, re-runs nothing)
goaly runs list
goaly runs show run-<id>

Tip: add .goaly/ to your repo's .gitignore (it holds run logs). Put repeated flags in a .goalyrc file once — per-project, or a personal ~/.goalyrc (e.g. { "autonomous": true }) — then just pass the goal. Exit codes: 0 DONE 1 FAILED / ABORTED 2 usage error

🔁 marks the only three places goaly loops back, each explicitly bounded: COMPILE re-authors on a correctable miss (--max-compile-retries), SEAL re-authors from your feedback on “revise” (--max-seal-revisions), and a red iteration re-runs the agent with the FAIL/veto fed back (--max-iterations, or until STUCK). Everything else runs once per pass — and the bar sealed at SEAL is never rewritten.

Verify cheap-and-ungameable first

The check is a ladder of rungs run top → bottom. Deterministic rungs (exit codes, tests) come before any LLM judge, and the ladder short-circuits on the first hard fail — so a red test never wastes a judge call. Try failing a rung:

det · 1npm test exits 0

tests / exit codes — ungameable, hardest to fake

det · 2npm run lint exits 0

static checks before any model is consulted

smoke · 3--smoke "node smoke.mjs" exits 0

actually runs the built artifact — a second ungameable check

judge · 4LLM quorum vs. the frozen rubric

only the fuzzy residual — best-of-N, with a confidence floor

Two gates, and the one flag that moves only one

Seal The bar

Your say over the success check, before the loop: approve it, give feedback to revise it, or reject. --autonomous skips this pause — the contract is still frozen and logged.

Sign-off The result

Every iteration, but only when the check passes. An independent reviewer is veto-only — it can stop a green from becoming DONE, never promote a red.

--autonomous

Moves Seal only. It skips the human pause, never the freeze and never verification. The two-key contract is identical either way.

Phased: a frozen plan of frozen contracts

--phased decomposes one big goal into an ordered plan of small sub-goals, runs each as its own frozen, two-key contract (with an internal checkpoint between phases so each diff stays small), and finishes with a cumulative acceptance contract on the original goal — so decomposition can't green a goal whose parts pass but whole doesn't. The plan is frozen + hashed + logged; re-planning is only the bounded, human-gated plan-Seal revise path.

The planner is a read-only LLM seam (like the compiler) — or supply the plan yourself with --plan-file. --max-phases caps the plan; --budget-tokens is the whole-run total; --autonomous auto-accepts the plan and each phase (still frozen + logged); --resume re-enters mid-plan without repeating completed phases.

Delta diffs: a flat judge prompt, a cumulative DONE

--delta-verify (default off) keeps the per-iteration judge prompt flat on a long run: after each continuation iteration goaly takes an internal checkpoint (the same private tree snapshot as --baseline — no commit), so the next iteration's judge reviews only that iteration's delta instead of the whole cumulative diff. The trust model is preserved by keeping the DONE decision cumulative: the deterministic rungs always run on the full working tree (they execute commands, not diffs — the ungameable key), and the terminal Sign-off approver is pinned to the run's start baseline, so it reviews the entire change — a violation smeared across iterations, where no single delta looks bad, is still caught. A checkpoint that can't be taken falls back to the full diff (never an empty one). It composes with --phased: per-iteration deltas feed the judge within a phase, while the approver baseline advances only at phase boundaries — so each phase's Sign-off still reviews that phase's whole cumulative diff.

Extending it

Adding a harness is one small module.

A "harness" is any coding agent you can drive headlessly. The orchestrator can't tell which one it called — so wrapping a new one changes nothing else in the system. You write one AgentCliCodec that holds everything goaly needs to know about that CLI, and register it with one line.

What you need to figure out about a CLI

01
How to run one headless turn

The non-interactive / print flag (-p, exec, --print). The prompt is usually an argv value or read from stdin.

02
How to read the result

The machine-readable output (--output-format json, --json). Note the shape — a whole-stdout JSON object, a JSONL stream, or text-only — and which field carries the answer.

03
How to resume the session

Where the session id appears in a fresh run, and which flag replays it next turn (--resume, --session-id, exec resume <id>). No resume → return a stable id and accept cold turns.

04
Per-turn streaming (optional)

If the CLI has a streaming mode, map each line onto the canonical AgentStreamEvent taxonomy so the live view works. No streaming → skip it; the run is unaffected.

The one interface you implement

interface AgentCliCodec {
  readonly name: string;
  readonly command: string;                 // the binary to spawn
  readonly fieldExtractor: FieldExtractor;     // where result / session / tokens live
  readonly streamExtractor: StreamEventExtractor; // per-turn → AgentStreamEvent (optional)
  harnessArgs(opts): string[];               // write-mode argv (drives edits)
  readonlyArgs(opts): string[];              // read-only argv (judge / approver)
  parse(stdout): AgentOutput | null;          // tolerant, never throws
  classify(input): HarnessRunResult;          // process outcome → status
}

The shared core does the heavy lifting — the tolerant JSON/JSONL parse, the subprocess dance (timeouts, output caps, process-group kill), the diff hashing, and verifier execution all live outside the codec. So your module is just this CLI's quirks, in one place. The same codec also powers the read-only LLM steps (judge / approver / compiler) for free.
