Goal orchestration

Loop a coding agent until the goal is verifiably met.

goaly runs any coding agent — Claude Code, Codex, Droid, your own — until a goal is actually done, against a frozen success contract it can't weaken mid-loop.

harness-agnostic/ zero-LLM control flow/ two keys for DONE/ replayable & resumable

The core problem

“Until the goal is achieved” must not collapse into
“until the agent weakens its own test.”

If an agent can regenerate its own success criterion each iteration, an autonomous loop degenerates into reward hacking. goaly's whole design exists to prevent that.

01
Compile once, then freeze

The success contract — an ordered ladder of checks + a rubric — is authored once, hashed, and frozen. No transition rewrites it; the contractHash is logged every iteration to prove the bar never moved.

02
Two independent keys

A run is DONE only when both keys turn: the frozen verifier passes and an independent approver doesn't veto. A worker can game one; getting both is much harder.

03
Fail closed, always

Any verifier, grader, approver, or adapter that errors or returns garbage becomes a FAIL / VETO — never a false green, never an unhandled throw.

Internal functionality

Deterministic stages, decision points, budgets.

Everything stochastic — running the agent, judging, approving — already happened before the control flow sees it. What's left is a pure, replayable state machine.

The state machine

Pure, synchronous, zero-LLM. Click a state to see the command it emits and the event that advances it.

DECIDE — the truth table, made interactive

Flip the inputs. The outcome follows the exact precedence of decide.ts — success is checked first, so a goal met on the last allowed iteration is DONE, never FAILED.

DONE
  1. ladder.pass && !veto → DONE
  2. stuck → ABORTED
  3. iteration ≥ max → FAILED
  4. !ladder.pass → CONTINUE
  5. ladder.pass && veto → CONTINUE

The verifier ladder

Cheapest & hardest-to-game first. Rungs run top → bottom and short-circuit on the first deterministic fail, so a red exit code never wastes an LLM judge call. Try failing a rung:

det · 1npm test exits 0

tests / exit codes — ungameable, confidence = 1

det · 2npm run lint exits 0

static checks before any model is consulted

judge · 3LLM quorum vs. the frozen rubric

only the fuzzy residual — temp 0, best-of-N, confidence floor

How the LLM judge is made “deterministic enough”4 techniques
  • Frozen rubric — criteria authored once; the bar can't drift to meet the work.
  • Temperature 0 + structured output — a Zod schema {pass, confidence, failing_criteria[]}, no free-form “looks good”.
  • Quorum, not one call — best-of-N majority (default 3) + a confidence floor (default 0.66).
  • Independent grader — fed goal + rubric + diff + verdicts, not the worker's self-justification.

Two gates, and one flag that moves only one

Gate A Contract

The human's say over the bar, before the loop. goaly prints the frozen contract and prompts [a]pprove / [f]eedback / [r]eject: approve starts the loop, feedback re-authors the contract from your free-text note and re-prompts (bounded by --max-gate-a-revisions, default 10), reject aborts. --autonomous skips the prompt: auto-accepted, but still frozen and logged loudly.

Gate B Result

Every iteration, but only when the ladder passes. The independent approver is veto-only — it can stop a green from becoming DONE, never promote a red. Defaults to reject on uncertainty.

--autonomous

Moves Gate A only. It skips the human pause, never the freeze and never verification. The two-key contract is identical in both modes.

Stuck detection & budget

Bail before the iteration cap with an actionable reason. Pure over the histories stored in loop state.

01
No-diff

The working tree is unchanged after a whole iteration → the agent is spinning.

02
Repeat-failure

The same normalized verifier failure N times in a row (default 3) → same wall.

03
Oscillation

The diff hash flip-flops A,B,A,B between two states → flip-flopping.

04
Budget

Token / wall-clock caps, metered by the driver — independent of iteration count.

Internal software architecture

One deep module. Four real seams.

The orchestrator (a pure reducer) owns all policy. A thin driver performs effects. Everything stochastic hides behind a value interface at four seams — each with a real and a fake implementation, so the orchestrator can't tell which it called.

imperative · the only thing that touches the world

Driver

Performs commands, feeds events back, persists write-ahead. Clock, budget, processes, disk — all live here.

pure · synchronous · zero-LLM by construction

Orchestrator — step(state, event) → [state, Command[]]

The whole COMPILE → Gate A → loop → DECIDE graph, iteration count, stuck bookkeeping. Returns no Promise, holds no adapters: it cannot perform an effect. That is what makes the run replayable from the log.

Parse at every seam (Zod, parse-don't-validate)validation

No path from the outside world into the reducer skips a parse. Branded ids are nominal (SessionId, DiffHash, RunId, ContractHash).

  • CLI args & run config → reject with a usage error
  • Harness stdout (hostile/partial) → never throw; map to truncated/crashed
  • Judge / approver output → drop unparseable (judge) / fail-closed (approver)
  • Run log on resume → reject a corrupt entry
Write-ahead log → replay & resumedurability

Each event is persisted before the state advances. Because the reducer is pure:

  • replay = fold the reducer over the event stream
  • resume = parse header + entries, replay-fold to reconstruct state (session id, iteration, stuck histories), then continue
  • inspect = goaly runs list / runs show <id> reuse the same replay-fold (read-only, re-runs nothing) so the rendered status/iterations match exactly what the Driver computed
  • no completed iteration is repeated; durability is at-least-once by design
  • logs live under .goaly/<runId>/, excluded from the tree hash

Quickstart

# install (puts goaly on your PATH)
make install
goaly help

# point at an existing test; approve the contract once
goaly run --goal "handle empty input" \
  --verify-cmd "npm test"

# longer goal from a file (or stdin); revise the contract at Gate A up to 3×
goaly run --goal-file ./GOAL.md \
  --generate --max-gate-a-revisions 3
cat GOAL.md | goaly run --goal - --generate
# let the agent author verification, run unattended
goaly run --goal "add a /health endpoint" \
  --generate --autonomous

# choose a harness, cap iters, set a budget, resume
goaly run --goal "…" --harness codex \
  --max-iterations 8 --budget-tokens 500000
goaly run --goal "…" --resume run-<id>

# model for the harness + a different model (and CLI) for the LLM steps
goaly run --goal "…" --model claude-opus-4-8 \
  --llm-model claude-sonnet-4-6 --llm-provider claude

# follow the loop step-by-step; structured rotated file at .goaly/<runId>/goaly.log
goaly run --goal "…" --verify-cmd "npm test" --log-level debug

# watch the agent's turns live — run AND LLM steps — tool calls, messages, tokens
goaly run --goal "…" --verify-cmd "npm test" --stream

# cap each step's subprocess (ms); a verify timeout fails closed (FAIL, never green)
goaly run --goal "…" --verify-cmd "npm test" \
  --harness-timeout-ms 900000 --verify-timeout-ms 60000

# put repeated wiring in .goalyrc once, then just pass the goal
# { "harness": "codex", "verify-cmd": "npm test", "autonomous": true, "max-iterations": 8 }
goaly run --goal "add a /health endpoint"
# …or point at an explicit profile that overrides .goalyrc
goaly run --goal "…" --config ./ci/goaly.ci.json

# inspect past runs (read-only — replays the run log, re-runs nothing)
goaly runs list
goaly runs show run-<id>

Exit codes — 0 DONE 1 FAILED / ABORTED 2 usage error

Extending it

Adding a harness is one deep codec.

A “harness” is any coding agent you can drive headlessly. The orchestrator can't tell which one it called — so wrapping a new one changes nothing else in the system. The seam stays this one method; the per-CLI substance is a single AgentCliCodec.

interface HarnessAdapter {
  readonly name: string;
  run(prompt: string, sessionId?: SessionId, onEvent?: AgentEventSink): Promise<HarnessRunResult>;
}  // onEvent: optional, opt-in streaming tap (issue #23)

type HarnessRunResult = {
  output: string;
  sessionId: SessionId;
  status: 'completed' | 'crashed' | 'truncated' | 'timeout';
  tokensUsed?: number;
};  // diffHash is computed by the shared Workspace — NOT the adapter

Behind that one method sits the real unit of work — an AgentCliCodec that holds all of one CLI's quirks in one place: its two argv dialects (harnessArgs for the write-mode agent, readonlyArgs for the judge/approver), its fieldExtractor + streamExtractor, and its classify status mapping. The generic AgentCliHarness consumes it for seam #1, and the read-only AgentCliLlmProvider consumes the same codec for the LLM steps — so there's no llm → harness coupling and a CLI's knowledge never scatters. Registering one is a codec module + one line in makeHarness.

The three things to figure out

01
Headless / print invocation

How do you run one non-interactive turn? Look for a print/exec/headless flag (-p, exec, --print). The prompt is usually an argv value or read from stdin.

02
Structured output

Find a machine-readable result (--output-format json, --json). Note the shape — a whole-stdout JSON object, a JSONL stream, or text-only — and capture a real sample.

03
Session resume

Where does the session id appear in a fresh run's output, and which flag replays it next turn (--resume, --session-id, exec resume <id>)? No resume → return a stable id and accept cold turns.

04
Per-turn streaming (optional — issue #23)

Is there a streaming output mode (--output-format stream-json, an always-JSONL --json)? If so, note which line carries each turn — message, tool call + result, usage — and map them onto the canonical AgentStreamEvent taxonomy. No streaming mode → skip it; the run is unaffected.

What run() must & must not docontract

MUST

  • Spawn headlessly with the prompt, resuming sessionId when provided.
  • Parse stdout tolerantly — partial output, crashes, and non-JSON are normal.
  • Thread the session id out so the next iteration resumes the same conversation.
  • Map the outcome to a status and return a Zod-parsed HarnessRunResult.
  • Never throw. Any failure becomes a non-completed status.

MUST NOT

  • Compute diffHash — the shared Workspace does, so stuck-detection works on any harness for free.
  • Run the verifier — verification is “run this command in the workspace,” identical everywhere.
  • Leak harness specifics — flag dialects, hook tricks stay inside run().
The mappings: stdout → fields, outcome → status, turns → eventsfield · status · stream

1 · Field mapping

A small FieldExtractor — (obj) => { text?, sessionId?, tokens? } — by your CLI's key names. The shared parseAgentOutput core owns the envelope walk (whole-object / amid-noise / JSONL), latches the first session id, keeps the last text-bearing line, and returns null on no text. Flat envelope? Reuse the shared flatExtractor and write nothing.

2 · Status mapping

killed for exceeding wall-clocktimeout
exited non-zerocrashed
exit 0, parseable, non-emptycompleted
exit 0 but empty / unparseabletruncated
the exec seam itself threwcrashed

3 · Stream mapping (optional — issue #23)

The streaming sibling of the field mapping. A StreamEventExtractor — (obj) => AgentStreamEvent[] — maps one JSONL line of your CLI's per-turn output onto the canonical, tool-neutral AgentStreamEvent taxonomy (session · message · reasoning · tool_use · tool_result · usage · done). The shared StreamTap buffers partial lines across chunks, Zod-validates every event at the seam, and forwards survivors to the live view / logger / embedder — fail-closed, so a throwing sink never touches the run. Emit the Anthropic agent-SDK envelope (claude-code, droid)? Reuse sdkStreamExtractor. Final-envelope only? flatStreamExtractor. The same mapping serves the read-only LLM steps too — write it once, both seams stream.

Register it — two tiny editsargs.ts · compose.ts 
// src/cli/args.ts — add the literal to the choice union + parser
export type HarnessChoice = 'claude-code' | 'codex' | 'droid' | 'fake' | 'myagent';

// src/cli/compose.ts — wire your codec into makeHarness() via the generic adapter
case 'myagent': return new AgentCliHarness(myagentCodec, opts);

Then add your codec to the shared adapter.contract.test.ts matrix — it runs every adapter through the same scenarios (success / non-zero / garbage / timeout / exec-throws) to prove it never throws and always returns a valid HarnessRunResult.

Optional: also run the LLM steps on itread-only provider

A harness drives the agent. The compiler / judge / approver are a separate, read-only seam — LlmProvider.complete(prompt) → {text, tokensUsed?} (the optional token count feeds the per-run spend report). Different interface on purpose: a judge must never edit the tree it's judging (Interface Segregation). But it reuses the codec you already wrote — its fieldExtractor and its readonlyArgs dialect (codex --sandbox read-only; droid omits --auto) — so wiring it is a makeLlmProvider() case + an LlmProviderChoice literal, the model cascade (--llm-model, per-step flags) threaded in for you. No read-only mode on your CLI → don't wire it as a provider.

The LLM steps stream too (issue #23): under --stream the compile / judge / approve turns surface in the live view just like the agent run, phase-tagged. They reuse the same StreamEventExtractor — the sink is wired into the provider at construction (so the Verifier / Approver seams stay clean), and the same canonical events flow out. One stream mapping, both seams.

Built for agents, by agents

goaly drives coding agents — and is built to be worked on by them.

The repo ships its own agent tooling: a skill that teaches an agent to onboard a new harness, and a hard set of invariants any contributor — human or model — must preserve.

🛠️

investigate-harness skill

A bundled agent skill that probes an unfamiliar CLI and emits a filled adapter. Read-only first (--help, --version), runs the agent only in a throwaway git repo, redacts session ids, and produces the field/flag/status mapping plus the adapter, the two registration edits, and tests.

The eight invariants every agent must preserveAGENTS.md
1Zero-LLM reducer. step/decide/stuck are pure & synchronous — no Promise, clock, IO, or adapter imports.
2Compile once, then freeze. The contract is authored once; no transition rewrites it.
3Two keys for DONE. Frozen verifier passes and the approver doesn't veto. Gate B runs only on a green ladder.
4Fail-closed everywhere. Any error or unparseable output → FAIL / VETO / crashed-run, never a green.
5--autonomous moves Gate A only. It auto-accepts but still freezes and logs the contract.
6Parse at every seam (Zod). Nothing reaches the reducer without a parse; ids are branded.
7Write-ahead + resume. Events append before state advances; resume is a replay-fold over the log.
8Stuck detection stays pure over loop histories: no-diff, repeat-failure, oscillation, budget.

Fakes-first, zero-IO TDD

The walking skeleton — domain → pure reducer → fakes → driver — is proven end-to-end with zero IO before any subprocess exists. Every seam has a fake, so the whole policy is table-testable.

Docs are part of “done” explicit check

Any change to the architecture, the public API, or functionality must update the README and this landing page in the same change — a definition-of-done gate, not optional. This page cites real states, commands, events, and defaults.
