What it is

Tell it the goal. It runs the agent until the goal is really met.

You point goaly at a goal and a way to check it. It runs your coding agent over and over — feeding back what's still wrong each time — and only stops when the work passes a fixed bar. The trick is that bar can't move: the agent can't quietly weaken its own test to "finish".

1 Freeze the bar

The success check is written once, hashed, and frozen before the loop starts. No later step can rewrite it — so "until it's done" can't collapse into "until the agent lowers the bar".

2 Two keys for DONE

A run is done only when both turn: the frozen check passes and an independent reviewer doesn't veto. Gaming one is possible; getting both is much harder.

3 Fail closed

Anything that errors, crashes, or returns garbage becomes a fail — never a false green. A broken check can never accidentally mark the run complete.

That's the whole point: correctness you can trust when the agent runs unattended. The control loop itself makes zero LLM calls — it's a small, deterministic state machine — so every run is replayable and resumable from a log. See how the loop works ↓

Get started

From install to your first verified run in a few minutes.

Before you start: Node ≥ 20 and git, plus at least one agent CLI on your PATH — claude, codex, droid, or pi. (No CLI yet? Use --harness fake to try the loop with zero network.)

01
Install it

Builds dist/ and puts goaly on your PATH.

# clone, then from the repo root:
make install        # == npm install -g .
goaly help
02
Run against a check you already have

Point at an existing test command. goaly freezes the contract, asks you to approve it once at Seal, then loops until npm test passes and the reviewer agrees.

goaly run --goal "make the parser handle empty input" \
  --verify-cmd "npm test"
03
Or just give it a goal — the agent writes the check too

The goal is a positional and run is optional. With no --verify-cmd, the LLM authors the verification (--generate) and checks the work, using Claude — all defaults. Add -d (alias --defaults) to run hands-off: it auto-accepts the Seal pause (the contract is still frozen and logged loudly).

# approve the contract once at Seal
goaly "add a /health endpoint returning 200"

# …or fully hands-off
goaly -d "add a /health endpoint returning 200"

Good to know: add .goaly/ to .gitignore (it holds run logs). Put repeated flags in a .goalyrc (per-project or ~/.goalyrc), then just pass the goal. Exit codes: 0 DONE 1 FAILED / ABORTED 2 usage

How it works

One compile, then a verified loop.

A single pure, zero-LLM reducer owns all the policy; a thin driver does the work it asks for. There are exactly two gates and three bounded retry edges — nothing else loops. Click any stage to see what it does.

🔁 marks the only three places goaly loops back, each explicitly bounded: COMPILE re-authors on a correctable miss, SEAL re-authors from your feedback on “revise”, and a red iteration re-runs the agent with the FAIL/veto fed back. Everything else runs once per pass — and the bar sealed at SEAL is never rewritten.

Verify cheap-and-ungameable first

The check is a ladder of rungs run top → bottom. Deterministic rungs (exit codes, tests) come before any LLM judge, and the ladder short-circuits on the first hard fail — so a red test never wastes a judge call. Try failing a rung:

det · 1npm test exits 0

tests / exit codes — ungameable, hardest to fake

det · 2npm run lint exits 0

static checks before any model is consulted

smoke · 3--smoke "node smoke.mjs" exits 0

actually runs the built artifact — a second ungameable check

judge · 4LLM quorum vs. the frozen rubric

only the fuzzy residual — best-of-N, with a confidence floor

Two gates, and the one flag that moves only one

Seal The bar

Your say over the success check, before the loop: approve it, give feedback to revise it, or reject. --autonomous skips this pause — the contract is still frozen and logged.

Sign-off The result

Every iteration, but only when the check passes. An independent reviewer is veto-only — it can stop a green from becoming DONE, never promote a red.

--autonomous

Moves Seal only. It skips the human pause, never the freeze and never verification. The two-key contract is identical either way.

Under the hood it's a deep module at every seam — the harness, the verifier ladder, the approver and the compiler each hide a lot of behaviour behind a small interface, so the policy is testable in one place and a new variant is one file. The full picture — the state machine, the DECIDE truth table, stuck detection, the four seams, and the write-ahead log that makes runs replayable — is in the architecture docs. What else you can turn on

Defaults get you a verified run with one command. These flags scale it to bigger goals, longer runs and tighter isolation — every one is documented in the README.

--phased Big goals

Decompose one large goal into a frozen plan of small, individually-verified sub-goals, then accept cumulatively on the original — so parts passing can't green a broken whole.

--delta-verify Long runs

Keep the per-iteration judge prompt flat by reviewing only each iteration's delta — while the DONE decision stays cumulative and ungameable.

--sandbox Isolation

Jail the agent and the verifier in an OS sandbox (bubblewrap / container): deny $HOME, control egress. One policy, many launchers.

--resume run-<id> Recover

Continue a crashed run from its write-ahead log — same session, same iteration count, no completed work repeated.

--stream Observe

Watch the agent's turns live — tool calls, messages, token counts — for the run and the LLM steps; persist it for offline replay.

--budget-tokens Spend

Cap the whole run by tokens or time; every run ends with a per-step spend report (add an approximate USD cost on request).

Extending it

Adding a harness is one small module.

A "harness" is any coding agent you can drive headlessly. The orchestrator can't tell which one it called — so wrapping a new one changes nothing else in the system. You write one AgentCliCodec that holds everything goaly needs to know about that CLI, and register it with one line. The shared core does the heavy lifting — the tolerant JSON/JSONL parse, the subprocess dance, diff hashing, and verifier execution all live outside your codec.

1 · Run one turn

The headless/print flag (-p, exec, --print) and how the prompt is passed.

2 · Read the result

The machine-readable output shape and which field carries the answer, the session id, and token usage.

3 · Resume the session

Where the session id appears and which flag replays it (--resume, --session-id, …). Per-turn streaming is optional.
